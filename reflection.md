The strongest thing I learned from this assignment is that prompt quality is a product decision, not just a writing exercise. Once the UI, persona switching, and API wiring were laid out clearly, it became obvious that each persona needed its own behavioral contract: voice, length, structure, and refusal boundaries all had to be explicit or the model would drift toward the same generic answer style. The few-shot examples were especially useful because they gave the model concrete patterns to imitate instead of vague adjectives like “friendly” or “smart.”

The GIGO principle showed up very quickly. When a prompt is shallow, the output becomes shallow; when the prompt is specific about audience, constraints, and response shape, the model becomes much more useful. In practice, that means the quality of the research matters as much as the quality of the interface. If the persona details are inaccurate or thin, the assistant may still sound polished, but it will not feel authentic or trustworthy.

What worked best was treating the app as a complete system rather than separate frontend and backend tasks. The persona switcher resetting the conversation made the experience cleaner, the suggestion chips lowered the friction for first-time use, and the typing indicator made the interaction feel alive. The separation between `src/prompts.js` and `prompts.md` also helped because it kept the runtime configuration and the documentation version in sync.

If I improved this further, I would spend more time on persona research and add stronger evaluation checks for prompt quality. I would also add conversation memory controls, better streaming responses, and a deployment pipeline so the app could be shipped with less manual setup. The biggest takeaway is that a chatbot feels credible only when the research, prompt design, and interface all reinforce the same personality.
